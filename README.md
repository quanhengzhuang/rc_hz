# rc_hz

## 需求
企业内部多个业务系统在关键事件发生时，需要调用外部系统供应商提供的 HTTP(S) API 进行通知。例如：
- 用户通过第三方广告系统引流并成功注册后，通知对应的广告系统
- 用户订阅付款成功后，通知 CRM 系统更改 Contact 状态
- 用户购买商品后，通知库存系统进行库存变更

不同供应商的 API：
- 请求地址不同
- Header / Body 格式不同

业务系统本身：
- 不需要关心外部 API 的返回值
- 只需确保通知请求能够被稳定、可靠地送达

## 思路
类似一个「事件消息 -> 单一订阅者」的消息队列，可以抽象出一个消息路由，叫 `MessageRouter`，可分为 pub、sub、server 三个视角来考虑。

pub 端:
- 企业内部业务系统的事件，如：用户注册成功、订阅付款成功、购买商品等
- pub 端发消息给 MessageRouter，得到成功回复即可
- 针对 pub 端的接口协议可用固定格式

sub 端：
- sub 端为各个外部系统，如广告系统、CRM 系统等
- sub 端的接口格式不同，所以最好有一层适配层，用于将 pub 端的消息转为其请求参数，并识别 pub 端的返回是否成功

MessageRouter Server 要做的事:
- 为 pub 端提供一个 https 接口
- 适配层（可配置），用于将 pub 端的消息转为 sub 端需要的格式，并处理返回值
- 可靠存储 pub 过来的消息，可使用消息队列
- 可靠投递给 sub 端的消息，此处需要考虑失败重试
